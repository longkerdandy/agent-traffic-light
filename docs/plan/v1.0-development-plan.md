# Agent Traffic Light — v1.0 Development Plan

## 1. Overview

Build a standalone local service that drives the AgentCoreLight v1 ESP32-C3 device from Kimi Code sessions.

The v1.0 system is intentionally simple: a single .NET process that exposes a small HTTP API, aggregates Kimi Code lifecycle events, chooses which active session controls the hardware, and writes the corresponding serial command to the ESP32.

The official delivery package in `reference/` is used only as reference for the serial protocol, LED effects, and event schema. We are rewriting the control server from scratch.

## 2. Goals

- **Single-process architecture**: web API + serial bridge in one .NET service.
- **Kimi Code support**: consume lifecycle events from Kimi Code.
- **Explicit light ownership**: a session only controls the hardware after the user enables it with a slash command.
- **Multi-session arbitration**: when several Kimi Code sessions have the light enabled, the service selects one winner.
- **Heartbeat keep-alive**: sessions send heartbeats so the service knows they are still open even when idle.
- **Native Windows service**: runs as a Windows service out of the box; falls back to systemd on Linux.
- **No TUI feedback in v1.0**: feedback is limited to the web dashboard and, optionally, system notifications.

## 3. Scope

### In scope

- .NET service (`AgentTrafficLightServer`) with HTTP API and SSE stream.
- Serial port auto-detection and direct ESP32 control via `System.IO.Ports.SerialPort`.
- Kimi Code integration via native lifecycle hooks and a Kimi plugin/skill.
- Web dashboard for status, sessions, and manual testing.
- Heartbeat keep-alive for session liveness.
- Light enable/disable arbitration across Kimi Code sessions.
- Windows and Linux (including WSL) support.
- Windows service deployment via `Microsoft.Extensions.Hosting.WindowsServices`.

### Out of scope for v1.0

- Claude Code, Codex CLI, and other agent integrations.
- macOS support.
- TUI feedback inside Kimi Code.
- BLE transport to the hardware.
- Persistent history database.
- Cloud or remote access.
- Firmware changes; we use the stock firmware and serial command set.

## 5. Components

### 5.1 Server Core

A single .NET process built with:

- ASP.NET Core Minimal APIs for the HTTP API.
- Server-Sent Events (SSE) for live dashboard updates.
- `System.IO.Ports.SerialPort` for hardware communication.
- `Microsoft.Extensions.Hosting` background services for serial port detection and reconnects.
- A small in-memory session store.
- `Microsoft.Extensions.Hosting.WindowsServices` for Windows service integration.

Responsibilities:

- Ingest agent events.
- Track session liveness via heartbeats.
- Compute the current hardware state.
- Write serial commands to the ESP32.
- Serve a static dashboard.

### 5.2 Session Store

Each session is identified by `session_id` and stores:

- `agent`: `kimi` (v1.0 only).
- `cwd`: working directory.
- `lastEvent`: most recent lifecycle event.
- `lastSeen`: timestamp of last event or heartbeat.
- `lightEnabled`: whether the user turned the light on for this session.
- `expiresAt`: computed from heartbeat TTL.

Sessions are removed when the heartbeat TTL expires.

### 5.3 Serial Controller

- Auto-detect the ESP32 COM port by scoring available ports.
- Open the port at 115200 baud with DTR/RTS low.
- Maintain the last sent command and only write when the state changes.
- Reconnect automatically if the port disconnects.
- Accept manual override from the dashboard for testing.

### 5.4 Agent Integrations

#### Kimi Code

- Ship a Kimi Code plugin or skill that:
  - registers `/agent-light:on`, `/agent-light:off`, and `/agent-light:status` slash commands;
  - declares lifecycle hooks for `SessionStart`, `UserPromptSubmit`, `PreToolUse`, `PostToolUse`, `PermissionRequest`, `Stop`, `StopFailure`, `SessionEnd`, etc.;
  - keeps the session alive via heartbeat while Kimi Code is running.

- Hook commands POST JSON payloads to the server.

**Heartbeat binding — two candidate approaches (feasibility TBD)**

The heartbeat source must stop emitting as soon as Kimi Code exits or is force-closed. Relying solely on `SessionEnd` and server TTL is not acceptable. Two candidate designs are under consideration; the final choice depends on Kimi Code's skill/runtime model.

- **Approach A (preferred): resident skill sends heartbeat from inside Kimi Code**  
  If a Kimi Code skill can run persistently inside the Kimi Code process, the skill starts a timer on `SessionStart` and sends `POST /heartbeat` directly. When Kimi Code exits or is killed, the skill's process space is destroyed and heartbeats stop immediately. This avoids cross-platform process-binding complexity and works across the WSL/Windows-host boundary.

- **Approach B (fallback): heartbeat subprocess bound to Kimi Code's PID**  
  If the skill/hook can only spawn short-lived processes, the `SessionStart` hook obtains the Kimi Code parent PID (via `process.ppid`) and passes it to a dedicated heartbeat subprocess. The subprocess monitors that PID with OS-native mechanisms and self-terminates when Kimi Code dies:
  - Linux / WSL2: `pidfd_open(pid)` + `poll()` (kernel-level, no PID-reuse race); fallback to periodic `kill(pid, 0)` polling.
  - Windows native: Job Object with `JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE`.

> **Why not server-side PID validation?** In common setups such as Kimi Code running inside WSL while the server runs on the Windows host, the server cannot see or validate WSL Linux PIDs, so server-side PID checks are not viable.

### 5.5 Dashboard

A minimal browser UI served from the server:

- Current hardware state and which session controls it.
- List of active sessions with light on/off toggle.
- Manual test buttons for each serial command.
- Event log.

### 5.6 Security and Access Model

v1.0 intentionally implements **no authentication, authorization, or input validation beyond basic parsing**. To keep this acceptable, the server must:

- Bind to `127.0.0.1` only (never `0.0.0.0`).
- Be reachable only by local processes, including Kimi Code running inside WSL via host localhost forwarding.
- Not be exposed to the local network or the public internet.

Any future remote-access or multi-user scenario requires adding authentication and hardening first.

### 5.7 Windows Service Hosting

On Windows the application is hosted as a native Windows service:

```csharp
var builder = Host.CreateApplicationBuilder(args);
builder.Services.AddWindowsService();
builder.Services.AddHostedService<TrafficLightHostedService>();
var host = builder.Build();
host.Run();
```

Installation is performed via `sc.exe` or a small installer wrapper:

```cmd
sc create AgentTrafficLight binPath= "C:\path\to\AgentTrafficLightServer.exe"
sc start AgentTrafficLight
```

The service runs under `LocalSystem` by default; if a user-specific serial port requires it, the installer can configure a custom service account.

On Linux the same executable runs under systemd with an equivalent unit file.

## 6. State Machine

The hardware understands these serial commands:

| Command | Meaning |
|---------|---------|
| `idle` | Green breathing |
| `thinking` | Red-yellow-green chase (fast) |
| `ai` | Red-yellow-green chase (soft) |
| `busy` | Yellow slow blink |
| `wait_confirm` | Yellow steady |
| `success` | Green steady 5s, then idle |
| `error` | Red fast blink |
| `off` | All LEDs off |

The server maps agent lifecycle events to one of these commands:

| Agent event | Command |
|-------------|---------|
| `SessionStart` | `idle` |
| `UserPromptSubmit` | `thinking` |
| `PreToolUse` | `busy` |
| `PostToolUse` | `ai` |
| `PermissionRequest` | `wait_confirm` |
| `Stop` | `success` |
| `StopFailure` | `error` |
| `SessionEnd` / `Interrupt` | `off` |

## 7. Multi-Session Arbitration

Only sessions with `lightEnabled === true` participate in arbitration.

Winner selection:

1. Filter to enabled sessions.
2. Sort by priority:
   - Higher state priority wins (`error` > `wait_confirm` > `success` > `busy` > `ai` > `thinking` > `idle` > `off`).
   - Same priority: most recently active session wins.
3. The winning session’s command is written to the ESP32.

If no session has the light enabled, the hardware state is `off`.

## 8. API Surface

All endpoints are **localhost-only**. The server binds to `127.0.0.1` and implements no authentication.

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `/hook` | POST | Receive agent lifecycle event |
| `/heartbeat` | POST | Keep a session alive |
| `/api/master` | POST | Claim or release master control for a session |
| `/api/status` | GET | Current state, winner, sessions |
| `/api/sessions` | GET | All active sessions |
| `/stream` | GET | SSE stream of state changes |
| `/` | GET | Dashboard HTML |

## 11. Risks and Open Questions

- Kimi Code hooks are Beta; schema and config location may change.
- Heartbeat binding: relying on a detached heartbeat process plus server TTL is unacceptable because Kimi Code may be force-closed. Approach A (resident skill) or Approach B (PID-bound subprocess) must be validated and selected. Server TTL remains only as a final safety net.
- Windows service account may need explicit permissions for the target COM port.
- Self-contained .NET publish produces a larger binary than a Node.js runtime install.
- Should the service auto-start on boot, or be launched on demand by the first hook?

## 12. Success Criteria

- A Kimi Code session can enable the light with `/agent-light:on`.
- Multiple enabled Kimi Code sessions are arbitrated to a single winner.
- Hardware accurately reflects `thinking`, `busy`, `wait_confirm`, `success`, and `error` states.
- Sessions stay alive during idle periods via heartbeat.
- The server installs and runs as a Windows service without manual process management.
