# v0.2 Hardware Interaction Plan

## Goal

Make the server able to drive the AgentCore-Light v1 hardware from a Windows host. v0.2 delivers a **BLE-only control layer** that talks to the device over **Bluetooth Low Energy**, plus a **minimal command-line test harness**. No web API, no dashboard, no session management, no agent integration, and no USB-serial transport.

> **Implementation note:** The original v0.2 plan included a pluggable Serial/BLE transport layer. After testing, the ESP32-C3 built-in USB-Serial/JTAG controller proved unreliable for ordinary serial writes from Windows applications (known `Write timeout` issue), so the serial layer was removed and the implementation switched to **BLE-only control**. This document reflects the current code, not the original multi-transport intent.

## Scope

### In scope for v0.2

- A reusable `ITrafficLightController` abstraction.
- `BleTrafficLightController` for BLE control via the vendor GATT characteristic.
- Configuration-driven BLE connection (`BleOptions`).
- A `TrafficLightState` enum + mapping to BLE JSON payloads.
- A console / hosted-service entry point that keeps the process alive and drives the light.
- A CLI test harness for sending individual commands and running simple demos.
- Unit tests with the BLE controller.
- **Hardware integration test on a Windows host**: compile a self-contained Windows x64 executable and run it against the real ESP32 device over BLE.
- A modified firmware build (`firmware/esp32_c3_traffic_light/`) that exposes the BLE GATT service required by Windows.

### Out of scope for v0.2

- HTTP API / endpoints (`/hook`, `/heartbeat`, `/api/status`, etc.).
- Web dashboard / SSE.
- Session store, multi-session arbitration, heartbeat TTL.
- Kimi / Claude / Codex agent integrations.
- Windows service / systemd packaging.
- USB-serial transport and serial port auto-detection.
- Persistent configuration beyond `appsettings.json`.

## High-Level Architecture

```text
┌─────────────────────────────────────────────┐
│  Console Test Harness / Hosted Service      │
│  - Parses commands or runs a demo loop      │
└──────────────┬──────────────────────────────┘
               │ ITrafficLightController
               ▼
┌─────────────────────────────────────────────┐
│  BleTrafficLightController                  │
│  - Scans / connects to AgentCoreLight       │
│  - Writes JSON {"status":"..."} to GATT     │
└──────────────┬──────────────────────────────┘
               │ BLE GATT write
               ▼
       ESP32-C3 AgentCoreLight
```

## Components

### 1. `ITrafficLightController` (`server/AgentTrafficLightServer/Services/ITrafficLightController.cs`)

Public abstraction used by the test harness:

```csharp
public interface ITrafficLightController
{
    TrafficLightState CurrentState { get; }
    bool IsConnected { get; }
    Task ConnectAsync(CancellationToken cancellationToken = default);
    Task DisconnectAsync(CancellationToken cancellationToken = default);
    Task SetStateAsync(TrafficLightState state, CancellationToken cancellationToken = default);
}
```

### 2. `TrafficLightState` & Command Mapping (`server/AgentTrafficLightServer/Models/TrafficLightState.cs`)

```csharp
public enum TrafficLightState
{
    Off,
    Idle,
    Thinking,
    Ai,
    Busy,
    WaitConfirm,
    Success,
    Error
}
```

Mapping:

| State | JSON payload |
|-------|--------------|
| `Off` | `{"status":"off"}` |
| `Idle` | `{"status":"idle"}` |
| `Thinking` | `{"status":"thinking"}` |
| `Ai` | `{"status":"ai"}` |
| `Busy` | `{"status":"busy"}` |
| `WaitConfirm` | `{"status":"wait_confirm"}` |
| `Success` | `{"status":"success"}` |
| `Error` | `{"status":"error"}` |

Legacy aliases (`writing` → `ai`, `running` → `busy`, `done` → `success`) are handled by the firmware, not the server.

### 3. `BleTrafficLightController` (`server/AgentTrafficLightServer/Services/BleTrafficLightController.cs`)

Responsibilities:

- Scan for the configured BLE device by MAC address (`Ble:DeviceAddress`) or device name (`AgentCore-Light`).
- Discover the vendor GATT service and characteristic.
- Convert `TrafficLightState` to a JSON payload `{"status":"<command>"}` and write it to the characteristic.
- Maintain `GattSession` to keep the connection alive.
- Deduplicate repeated state requests.
- Implement `IAsyncDisposable`.

The controller also retries service discovery when Windows returns `ERROR_NOT_READY (0x80070016)` and waits for the GATT session to become active before marking itself connected.

Configuration section:

```json
{
  "Ble": {
    "DeviceAddress": "9C:CC:01:65:6E:72",
    "DeviceName": "AgentCore-Light",
    "ServiceUuid": "12345678-1234-5678-1234-56789abcdef0",
    "CharacteristicUuid": "12345678-1234-5678-1234-56789abcdef1",
    "ScanTimeoutMs": 10000,
    "ReconnectIntervalMs": 5000
  }
}
```

### 4. Console Test Harness (`server/AgentTrafficLightServer/Services/TestHarnessHostedService.cs`)

`Program.cs` builds a generic host (`Host.CreateApplicationBuilder`) and registers `TestHarnessHostedService` as a hosted service. The harness:

- Reads configuration from `appsettings.json`.
- Registers `BleOptions` and `ITrafficLightController -> BleTrafficLightController`.
- Presents a simple interactive menu:
  ```text
  Agent Traffic Light Test Harness
  Commands: idle thinking ai busy wait_confirm success error off demo quit
  >
  ```
- Sends the typed command to the hardware over BLE.
- `demo` cycles through states every 2 seconds until a key is pressed.
- `quit` cleanly disposes the controller and exits.
- Works with both interactive and redirected stdin.

### 5. Event Async Helper (`server/AgentTrafficLightServer/Utils/EventAsync.cs`)

Small utility used by the BLE controller to await Windows runtime events (`TypedEventHandler` / `EventHandler`) with cancellation and optional predicates.

## Firmware

The project now uses a custom PlatformIO firmware build under `firmware/esp32_c3_traffic_light/` instead of the stock `.ino`. Key changes from the delivery package:

- Adds a BLE GATT service (`12345678-1234-5678-1234-56789abcdef0`) and writable characteristic (`...def1`).
- Parses incoming JSON payloads and extracts `"status"`.
- Configures BLE Just Works pairing so Windows 10/11 can authenticate.
- Uses PWM LED effects matching the state table above.
- Sets yellow LED pin to **GPIO2** (verified on delivered hardware).

The original reference firmware remains read-only under `reference/`.

## Windows Host Testing Workflow

v0.2 hardware testing is done over **BLE** on Windows.

Prerequisites:

- The ESP32 AgentCoreLight device is flashed with the **custom firmware** (`firmware/esp32_c3_traffic_light/.pio/build/esp32-c3/firmware.bin`).
- The Windows host has Bluetooth enabled.
- The BLE MAC address of the device is known (visible in a BLE scanner such as nRF Connect).

Steps:

1. **Publish a Windows x64 self-contained executable from WSL** (or any development machine):
   ```bash
   dotnet publish server/AgentTrafficLightServer/AgentTrafficLightServer.csproj \
     -c Release \
     -r win-x64 \
     --self-contained true \
     -p:PublishSingleFile=true \
     -o ./artifacts/win-x64
   ```
2. **Copy `artifacts/win-x64/AgentTrafficLightServer.exe` to Windows** and place `appsettings.json` next to it.
3. **Edit `appsettings.json`**:
   ```json
   {
     "Ble": {
       "DeviceAddress": "9C:CC:01:65:6E:72",
       "DeviceName": "AgentCore-Light",
       "ServiceUuid": "12345678-1234-5678-1234-56789abcdef0",
       "CharacteristicUuid": "12345678-1234-5678-1234-56789abcdef1"
     }
   }
   ```
4. **Run the executable from Windows PowerShell/CMD**:
   ```cmd
   AgentTrafficLightServer.exe
   ```
5. Use the interactive menu to send commands and observe the hardware.

1. Publish a self-contained Windows x64 executable.
2. On the Windows host, run the executable with the ESP32 connected over BLE.
3. Manually verify each canonical command produces the expected light effect.

## Testing Strategy

- **Unit tests**: state extension mapping; BLE controller construction, connection guard, and deduplication; `EventAsync` helper behavior.
- **BLE validation**: use a generic BLE client (e.g. nRF Connect on a phone) to write `{"status":"busy"}` to the vendor characteristic and confirm the light changes.
- **Hardware integration test**: performed manually on a Windows host by running the published `AgentTrafficLightServer.exe` over BLE against the real ESP32 device. No automated hardware integration test is required in v0.2.

## Risks

- The ESP32-C3 built-in USB-Serial/JTAG controller is unreliable for ordinary serial writes from Windows (`Write timeout`). This is mitigated by using BLE only.
- WSL2 is used for development and unit tests only; **hardware testing is done on the Windows host** via a published self-contained executable.
- No hardware connected → unit tests still pass; hardware validation is a manual Windows host step.
- Windows BLE permissions: the first run may require administrator privileges or explicit Bluetooth consent in Windows Settings.

## Success Criteria

On a Windows host, with the ESP32 AgentCoreLight device running the custom firmware and advertising as `AgentCore-Light`, run the published `AgentTrafficLightServer.exe` over BLE and see the hardware respond to `idle`, `thinking`, `busy`, `success`, and `error` commands sent from the keyboard.
