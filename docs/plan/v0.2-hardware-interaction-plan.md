# v0.2 Hardware Interaction Plan

## Goal

Make the server able to drive the AgentCore-Light v1 hardware from a Windows host. v0.2 delivers a **pluggable control layer** that can talk to the device over **USB serial** or **BLE**, plus a **minimal command-line test harness**. No web API, no dashboard, no session management, and no agent integration yet.

The final v0.2 implementation uses **BLE as the primary transport on Windows** because the ESP32-C3 built-in USB-Serial/JTAG controller is unreliable for ordinary serial writes from Windows applications (known `Write timeout` issue). The serial layer is kept as a fallback and for non-Windows hosts.

## Scope

### In scope for v0.2

- A reusable `ITrafficLightController` abstraction.
- `SerialController` + `TrafficLightController` for USB serial control.
- `BleTrafficLightController` for BLE control via the vendor GATT characteristic.
- Configuration-driven transport selection (`Serial` or `Ble`).
- A `TrafficLightState` enum + mapping to ESP32 serial commands / BLE JSON payloads.
- A simple console / hosted-service entry point that keeps the process alive and drives the light.
- A CLI test harness for sending individual commands and running simple demos.
- Unit tests with mocked controllers.
- **Hardware integration test on a Windows host**: compile a self-contained Windows x64 executable and run it against the real ESP32 device over BLE.

### Out of scope for v0.2

- HTTP API / endpoints (`/hook`, `/heartbeat`, `/api/status`, etc.).
- Web dashboard / SSE.
- Session store, multi-session arbitration, heartbeat TTL.
- Kimi / Claude / Codex agent integrations.
- Windows service / systemd packaging.
- Persistent configuration beyond `appsettings.json`.
- Custom firmware builds (the stock firmware is used unchanged).

## High-Level Architecture

```text
┌─────────────────────────────────────────────┐
│  Console Test Harness / Hosted Service      │
│  - Parses commands or runs a demo loop      │
└──────────────┬──────────────────────────────┘
               │ ITrafficLightController
               ▼
┌─────────────────────────────────────────────┐
│  TrafficLightController                     │
│  - Decides which command to send            │
│  - Tracks current hardware state            │
└──────────────┬──────────────────────────────┘
               │ serial command string
               ▼
┌─────────────────────────────────────────────┐
│  SerialController                           │
│  - Auto-detects ESP32 COM port              │
│  - Opens/closes System.IO.Ports.SerialPort  │
│  - Writes newline-terminated commands       │
│  - Reconnects on disconnect                 │
└──────────────┬──────────────────────────────┘
               │ USB serial @ 115200
               ▼
        ESP32-C3 Traffic Light

┌─────────────────────────────────────────────┐
│  BleTrafficLightController                  │
│  - Connects to AgentCore-Light over BLE     │
│  - Writes JSON {"status":"..."} to GATT     │
└──────────────┬──────────────────────────────┘
               │ BLE GATT write
               ▼
        ESP32-C3 Traffic Light
```

The actual transport is selected at runtime through `TrafficLight:Transport` in `appsettings.json`.

## Components

### 1. SerialController (`server/AgentTrafficLightServer/Services/SerialController.cs`)

Responsibilities:

- Detect the best available serial port using a scoring heuristic:
  - Port friendly name/description contains `esp32`, `usb`, `serial`, `jtag`.
  - Exclude Bluetooth and pseudo-terminal ports.
  - Prefer ports that respond to a probe (optional in v0.2).
- Open the selected port at the configured baud rate (default 115200) with DTR/RTS low.
- Provide `WriteAsync(string command)` that appends `\n` and writes to the port.
- Maintain `IsConnected` state and expose a reconnect loop.
- Implement `IDisposable` / `IAsyncDisposable`.
- Support manual port override via configuration (`Serial:Port = "COM3"` or `/dev/ttyUSB0`).

Configuration section:

```json
{
  "Serial": {
    "Port": "auto",
    "BaudRate": 115200,
    "DtrEnable": false,
    "RtsEnable": false,
    "ReconnectIntervalMs": 2000
  }
}
```

### 2. TrafficLightState & Command Mapping (`server/AgentTrafficLightServer/Models/TrafficLightState.cs`)

```csharp
public enum TrafficLightState
{
    Off,
    Idle,
    Thinking,
    Ai,
    Busy,
    WaitConfirm,
    Success,
    Error
}
```

Mapping:

| State | Serial command |
|-------|----------------|
| `Off` | `off` |
| `Idle` | `idle` |
| `Thinking` | `thinking` |
| `Ai` | `ai` |
| `Busy` | `busy` |
| `WaitConfirm` | `wait_confirm` |
| `Success` | `success` |
| `Error` | `error` |

Legacy aliases (`writing` → `ai`, `running` → `busy`, `done` → `success`) are **not** needed in v0.2; the server emits canonical commands only.

### 3. TrafficLightController (`server/AgentTrafficLightServer/Services/TrafficLightController.cs`)

Responsibilities:

- Accept `SetStateAsync(TrafficLightState state)` calls.
- Only write to the serial port when the requested state differs from the last sent state.
- Convert state to command string and delegate to `SerialController`.
- Expose `CurrentState` for inspection.

Interface:

```csharp
public interface ITrafficLightController
{
    TrafficLightState CurrentState { get; }
    Task SetStateAsync(TrafficLightState state, CancellationToken cancellationToken = default);
}
```

### 4. BleTrafficLightController (`server/AgentTrafficLightServer/Services/BleTrafficLightController.cs`)

Responsibilities:

- Connect to a paired or known BLE device by MAC address (`Ble:DeviceAddress`).
- Discover the vendor GATT service and characteristic.
- Convert `TrafficLightState` to a JSON payload `{"status":"<command>"}` and write it to the characteristic.
- Reconnect automatically if the connection drops.

When no `DeviceAddress` is configured, the controller can scan for the configured `DeviceName` (`AgentCore-Light` by default).

Configuration section:

```json
{
  "Ble": {
    "DeviceAddress": "9C:CC:01:65:6E:72",
    "DeviceName": "AgentCore-Light",
    "ServiceUuid": "12345678-1234-5678-1234-56789abcdef0",
    "CharacteristicUuid": "12345678-1234-5678-1234-56789abcdef1",
    "ScanTimeoutMs": 10000,
    "ReconnectIntervalMs": 5000
  }
}
```

### 5. Console Test Harness (`server/AgentTrafficLightServer/Program.cs` for v0.2)

Replace the empty web host with a console application that:

- Reads configuration from `appsettings.json`.
- Registers `SerialController`, `TrafficLightController`, and `BleTrafficLightController` in the DI container.
- Selects `ITrafficLightController` based on `TrafficLight:Transport`.
- Starts a background service that:
  - Presents a simple interactive menu:
    ```text
    Agent Traffic Light Test Harness
    Commands: idle thinking ai busy wait_confirm success error off demo quit
    >
    ```
  - Sends the typed command to the hardware via the selected transport.
  - `demo` cycles through states every 2 seconds until a key is pressed.
  - `quit` cleanly disposes the controller and exits.

This keeps v0.2 as a plain console app. ASP.NET Core hosting is **not** used yet.

### 7. Windows Host Testing Workflow

Because the ESP32-C3 built-in USB-Serial/JTAG controller is unreliable for ordinary serial writes from Windows, v0.2 hardware testing is done over **BLE**.

Prerequisites:

- The ESP32 traffic light is flashed with the **stock firmware** (`firmware.bin`).
- The Windows host has Bluetooth enabled.
- The BLE MAC address of the device is known (visible in a BLE scanner such as nRF Connect).

Steps:

1. **Publish a Windows x64 self-contained executable from WSL** (or any development machine):
   ```bash
   dotnet publish server/AgentTrafficLightServer/AgentTrafficLightServer.csproj \
     -c Release \
     -r win-x64 \
     --self-contained true \
     -p:PublishSingleFile=true \
     -o ./artifacts/win-x64
   ```
2. **Copy `artifacts/win-x64/AgentTrafficLightServer.exe` to Windows** and place `appsettings.json` next to it.
3. **Edit `appsettings.json`**:
   ```json
   {
     "TrafficLight": { "Transport": "Ble" },
     "Ble": {
       "DeviceAddress": "9C:CC:01:65:6E:72",
       "DeviceName": "AgentCore-Light",
       "ServiceUuid": "12345678-1234-5678-1234-56789abcdef0",
       "CharacteristicUuid": "12345678-1234-5678-1234-56789abcdef1"
     }
   }
   ```
4. **Run the executable from Windows PowerShell/CMD**:
   ```cmd
   AgentTrafficLightServer.exe
   ```
5. Use the interactive menu to send commands and observe the hardware.

The serial path (`TrafficLight:Transport = "Serial"`) remains available for non-Windows hosts or for devices connected through an external USB-UART bridge.

### 5. Serial Port Scorer (`server/AgentTrafficLightServer/Services/SerialPortScorer.cs`)

Pure helper for ranking ports:

```csharp
public static class SerialPortScorer
{
    public static string? BestPort(IEnumerable<string> portNames, IEnumerable<string>? candidates = null);
}
```

Rules:

- Score each port by matching keywords in its friendly name/description.
- Exclude ports whose name/description contains `bluetooth`, `bt`, `rfcomm`.
- Return the highest-scoring port, or `null` if none match.

Platform notes:

- **Windows (primary hardware test target)**: use `System.IO.Ports.SerialPort.GetPortNames()` + WMI (`System.Management`) or registry for descriptions.
- Linux / WSL: parsing `/dev/serial/by-id/*` or `/dev/ttyS*` is supported, but **real hardware testing happens on Windows** in v0.2.

For v0.2, the scorer must work well on Windows because that is where the hardware is exercised.

## Development Phases

### Phase 1 — SerialController + scorer

1. Add `System.IO.Ports` package reference to the server project.
2. Implement `SerialPortScorer` with unit tests.
3. Implement `SerialController` with open/close/write/reconnect logic.
4. Add configuration options to `appsettings.json`.

### Phase 2 — TrafficLightState + TrafficLightController

1. Add `TrafficLightState` enum and extension method `ToSerialCommand()`.
2. Implement `TrafficLightController` with state deduplication.
3. Add unit tests for the controller using a fake `ISerialController`.

### Phase 3 — Console test harness

1. Convert `Program.cs` from empty web host to console app host.
2. Wire up DI: `IConfiguration`, `SerialController`, `TrafficLightController`.
3. Implement interactive menu + demo loop.
4. Test locally with or without hardware connected.

### Phase 4 — Windows host integration test

1. Publish a self-contained Windows x64 executable:
   ```bash
   dotnet publish server/AgentTrafficLightServer/AgentTrafficLightServer.csproj -c Release -r win-x64 --self-contained true -p:PublishSingleFile=true -o ./artifacts/win-x64
   ```
2. On the Windows host, run `artifacts/win-x64/AgentTrafficLightServer.exe` with the ESP32 connected.
3. Manually verify each canonical command produces the expected light effect.
4. Document the COM port auto-detection result and any manual override needed.

## API Surface (internal, no HTTP)

| Type | Member | Purpose |
|------|--------|---------|
| `ISerialController` | `bool IsConnected { get; }` | Connection status |
| `ISerialController` | `Task WriteAsync(string command, ...)` | Send a command |
| `ISerialController` | `ValueTask DisposeAsync()` | Cleanup |
| `ITrafficLightController` | `TrafficLightState CurrentState { get; }` | Last sent state |
| `ITrafficLightController` | `Task SetStateAsync(TrafficLightState, ...)` | Change light state |
| `TrafficLightController` | class | Serial transport implementation |
| `BleTrafficLightController` | class | BLE transport implementation |
| `TrafficLightState` | enum | Canonical state model |

## Testing Strategy

- **Unit tests**: mock `ISerialController` with a fake in-memory buffer; verify `TrafficLightController` only writes on state change and emits correct commands.
- **SerialPortScorer tests**: provide fixed port name/description lists; assert ranking and exclusion behavior.
- **BLE validation**: use a generic BLE client (e.g. nRF Connect on a phone) to write `{"status":"busy"}` to the vendor characteristic and confirm the light changes.
- **Hardware integration test**: performed manually on a Windows host by running the published `AgentTrafficLightServer.exe` over BLE against the real ESP32 device. No automated hardware integration test is required in v0.2.

- [ ] `BleTrafficLightController` connects to the device and writes JSON payloads that change the light state.

- [x] Stock firmware is used unchanged; no custom firmware is required.

## Risks

- The ESP32-C3 built-in USB-Serial/JTAG controller is unreliable for ordinary serial writes from Windows (`Write timeout`). The v0.2 Windows path therefore uses BLE instead of USB serial.
- WSL2 is used for development and unit tests only; **hardware testing is done on the Windows host** via a published self-contained executable.
- System.IO.Ports on Linux/WSL requires `libicu` for globalization and may still need `libudev` for serial enumeration. The serial path is not exercised on Windows in v0.2.
- No hardware connected → unit tests still pass; hardware validation is a manual Windows host step.
- Invariant globalization mode (local WSL setup) may surface `NETSDK1188` warnings again; already suppressed in test project.
- Windows BLE permissions: the first run may require administrator privileges or explicit Bluetooth consent in Windows Settings.

## Success Criteria

On a Windows host, with the ESP32 traffic light running the stock firmware and advertising as `AgentCore-Light`, run the published `AgentTrafficLightServer.exe` over BLE and see the hardware respond to `idle`, `thinking`, `busy`, `success`, and `error` commands sent from the keyboard. The serial path remains available as a fallback for non-Windows hosts.
