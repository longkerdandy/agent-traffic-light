# v0.4 Blazor Dashboard Plan

## 1. Goal

Add a browser-based dashboard to `AgentSignalBridge` using **Blazor Server**. The dashboard displays real-time hardware state, active agent sessions, and a stream of recent lifecycle events, and provides manual controls for local testing. It reuses the existing server-side services (`IAgentStore`, `IAgentCoreLightManager`, `StateChangeNotifier`) and is served from the same ASP.NET Core host at `/`.

## 2. Scope

### In scope for v0.4

- Blazor Server integration into the existing ASP.NET Core host.
- Single-page dashboard at `/` with the layout described below.
- Real-time updates via `StateChangeNotifier` (server-side channel) pushed to the Blazor circuit.
- Read-only views: current state, active sessions, event log.
- Manual controls for local testing: send a `SessionStart`, `UserPromptSubmit`, `PreToolUse`, `PostToolUse`, `PermissionRequest`, `Stop`, `StopFailure`, `SessionEnd`, or `Disconnect` event for a chosen agent, and claim/release master control.
- Responsive layout using plain HTML/CSS (no external UI framework dependency).

### Out of scope for v0.4

- Authentication or authorization.
- Persistent history / database storage.
- Multiple pages or routing beyond `/`.
- Mobile-specific optimizations.
- External UI frameworks (MudBlazor, Radzen, Bootstrap, etc.).

## 3. Page Layout

The dashboard is a single Blazor page rendered at `/`. The layout is a fixed header plus a two-column grid for the main panels on desktop, collapsing to a single column on narrow viewports. The design separates **agents** from **hardware** so that future versions can support multiple hardware devices without reworking the dashboard.

```text
│  Agent Signal Bridge Dashboard               [● Connected]      │

│  Active Agents            │  │  Active Hardware          │
│                           │  │                           │
│  Agent    Event   Master  │  │  Name      State  Master  │
│  ───────────────────────  │  │  ───────────────────────  │
│  agent-1  Thinking  ●     │  │  AgentCore  Idle   agent-1│
│  agent-2  Idle      ○     │  │  ...                      │
│  ...                      │  │                           │
│                           │  │  Connected: Yes           │
│                           │  │  Last updated: 12:34:56   │
└───────────────────────────┘  └───────────────────────────┘
┌─────────────────────────────────────────────────────────────────┐
│  Event Log                                                      │
│  ...                                                            │
└─────────────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────────────┐

```

### Section details

#### Header
- Left: project name and "Dashboard" label.
- Right: aggregate connection status indicator (green when at least one hardware driver is connected, red otherwise).

#### Active Agents (left column)
- Table with columns: Agent ID, Agent Name, Last Event, Master, Last Seen.
- Master row highlighted.
- Updates automatically when sessions are created, updated, expired, or when master changes.
- This panel is about **who** is connected and who owns the master role.

#### Active Hardware (right column)
- Table with columns: Hardware Name, State/Command, Master Agent.
- For v0.4 there is one row for the single AgentCore-Light BLE device.
- Also shows:
  - **Connected**: whether the driver is currently connected.
  - **Last Updated**: timestamp of the last command or state change.
- This panel is about **what** the hardware is doing, designed so that additional hardware devices can be added as rows in future versions.

#### Event Log (bottom row)
- Scrollable list of recent lifecycle events and state changes.
- Each entry: timestamp, agent ID, hardware name, event/command description.
- Maximum 100 entries; older entries are discarded.

#### Manual Controls (bottom row)
- Input fields: Agent ID and Agent Name.
- Buttons to send canonical `AgentEvent` values to `/hook`.
- Buttons to claim or release master control via `/api/master`.
- These controls are intended for local development and manual validation only.

## 5. Components

### 5.1 Blazor Server Hosting

- Add `AddServerSideBlazor()` to the service collection.
- Map the Blazor hub with `MapBlazorHub()`.
- Map the fallback page with `MapFallbackToPage("/_Host")`.
- Add `_Host.cshtml` and a root `_Layout.cshtml` (or use the default Blazor host pattern).
- Serve the dashboard at `/`.

### 5.2 Dashboard Page (`Dashboard.razor`)

The single Blazor component that composes the layout:
- Injects `IAgentStore`, `IAgentCoreLightManager`, and a dashboard-specific state service.
- Renders the four sections described in the layout.
- Implements `IDisposable` to unsubscribe from state changes when the circuit disconnects.

### 5.3 Dashboard State Service

A scoped or singleton service (`DashboardStateService`) that:
- Subscribes to `StateChangeNotifier.ReadAllAsync()`.
- Maintains a bounded event log (max 100 entries).
- Invokes `StateHasChanged()` on registered Blazor components when state changes.
- Exposes helper methods for the manual controls to call the existing HTTP endpoints (`/hook`, `/api/master`) using `HttpClient`.

### 5.4 Manual Controls

- Razor event handlers call `DashboardStateService.SendEventAsync(...)` and `DashboardStateService.SetMasterAsync(...)`.
- These methods internally use `HttpClient` to post to the local endpoints, keeping the dashboard decoupled from direct store mutations.

## 6. API and Service Interactions

The dashboard consumes the same v0.3 endpoints that external agent clients use:

| Dashboard Action | Endpoint | Purpose |
|------------------|----------|---------|
| Refresh state on load | `GET /api/status` | Current command, connection, master |
| Refresh sessions on load | `GET /api/sessions` | Active sessions table |
| Real-time updates | `StateChangeNotifier` | Push events via Blazor circuit |
| Send test event | `POST /hook` | Manual lifecycle event injection |
| Claim/release master | `POST /api/master` | Manual master arbitration |

## 7. Development Phases

### Phase 1 — Blazor Server skeleton

1. Add Blazor Server framework reference and services (`AddServerSideBlazor`).
2. Create `_Host.cshtml`, `_Layout.cshtml`, `App.razor`, and `Routes.razor`.
3. Configure endpoint routing: `MapBlazorHub`, `MapFallbackToPage("/_Host")`.
4. Verify that `/` serves a placeholder Blazor page.
5. Add a basic integration test that asserts the dashboard page returns 200.

### Phase 2 — Layout and static data

1. Create `Dashboard.razor` with the layout structure.
2. Inject `IAgentStore` and `IAgentCoreLightManager`.
3. Render Current State, Active Sessions, Event Log, and Manual Controls sections.
4. Populate static data on component initialization.
5. Add unit tests for layout rendering using bUnit.

### Phase 3 — Real-time updates

1. Create `DashboardStateService` that subscribes to `StateChangeNotifier`.
2. Wire the service into `Dashboard.razor` and trigger `StateHasChanged` on updates.
3. Maintain a bounded event log in the service.
4. Add tests verifying that state changes propagate to the component.

### Phase 4 — Manual controls

1. Implement `HttpClient` calls for `/hook` and `/api/master` in `DashboardStateService`.
2. Wire the manual control buttons to these methods.
3. Add integration tests that press a button and verify the store/hardware driver receives the event.

### Phase 5 — Styling and validation

1. Add a minimal CSS file for layout, colors, and responsive collapse.
2. Run `dotnet format` and `dotnet format style`.
3. Run the full test suite.
4. Manual validation: open `/` in a browser, send events via dashboard, and verify the UI updates.

## 8. Testing Strategy

- **Unit tests**: bUnit tests for `Dashboard.razor` rendering and event handling with mocked services.
- **Integration tests**: ASP.NET Core TestServer tests that request `/_Host` and verify the dashboard returns a 200, and that manual controls result in store/driver changes.
- **Manual test**: open the dashboard in a browser, claim master with one agent, send events, and confirm real-time updates.

## 9. Configuration

No new configuration sections are required. The dashboard uses the existing `ServerOptions` host and port.
